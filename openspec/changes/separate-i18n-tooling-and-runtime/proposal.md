## Why

The current i18n branch mixes tooling ownership, runtime helpers, and non-package catalogs across `./i18n`, repository-root catalog files, and scattered `src/` modules. This makes the branch harder to rebase and has already produced duplicated runtime assets such as `commitDisplayLocalization.generated.ts`, which mirrors the JSON catalogs instead of treating them as the source of truth.

## What Changes

- Establish a strict directory boundary: `./i18n` is for extraction, catalog generation, locale sync, pending reports, glossary/passthrough policy, and other translation-maintenance tooling; `src/i18n` is for extension/runtime code and runtime-consumed catalogs.
- Move non-manifest runtime catalog ownership into `src/i18n/<surface>/`, keeping source English catalogs and locale catalogs together for surfaces that are not constrained by VS Code manifest NLS rules.
- Keep `package.nls.json` and `package.nls.zh-cn.json` at the repository root because VS Code manifest localization requires that layout.
- Replace commit-display generated TypeScript catalog duplication with direct runtime consumption of the commit-display JSON catalogs under `src/i18n/commitDisplay`.
- Consolidate runtime i18n helpers and adapters under `src/i18n`, rather than leaving new branch-local runtime code scattered across `src/system/-webview`, `src/webviews`, and surface implementation folders.
- Introduce shared `./i18n` tooling/report infrastructure for catalog diffing, locale sync, base-ref reads, stable report writing, accepted English passthrough values, and reusable zh-CN glossary behavior.
- Keep extractors surface-specific, but route their extracted English catalogs through the shared tooling post-processing and reporting workflow.

## Capabilities

### New Capabilities

- `i18n-tooling-workflow`: Defines shared branch-local tooling for extracted catalogs, zh-CN sync, pending translation reports, accepted passthrough values, and reusable translation-maintenance policy.

### Modified Capabilities

- `i18n-governance`: Clarifies the required boundary between `./i18n` tooling and `src/i18n` runtime assets/code.
- `commit-display-localization`: Moves commit-display runtime catalog ownership to `src/i18n/commitDisplay` and removes the duplicated generated TypeScript catalog as the runtime source.
- `webview-html-localization`: Moves webview catalog ownership for non-package webview HTML localization to `src/i18n/webviews` while keeping generated HTML template artifacts in `dist/webviews`.
- `webview-runtime-localization`: Centralizes webview runtime i18n helpers under `src/i18n/webviews` and keeps extraction/report tooling under `./i18n`.

## Impact

- Affected tooling: `./i18n/package`, `./i18n/webviews`, `./i18n/commitDisplay`, and new shared tooling modules under `./i18n/shared`.
- Affected runtime code: commit-display runtime adapter, webview host/client localization helpers, and catalog import/read paths under `src/i18n`.
- Affected artifacts: `commitDisplay.nls*.json`, `webviews.nls*.json`, `commitDisplayLocalization.generated.ts`, and report outputs generated by `report:*:pending` scripts.
- Affected build/package behavior: package manifest NLS remains root-owned; non-package runtime catalogs become runtime assets under `src/i18n` and must remain available in desktop and web builds.
